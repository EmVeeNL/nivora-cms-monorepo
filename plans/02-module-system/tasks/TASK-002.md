## Summary
Build the module registry generator — a `tsx`-executable script that discovers all `@nivora-cms/*` npm packages and `src/modules/*` locals, applies extend/merge logic, and emits `src/core/module-registry.gen.ts`. Wrap it in a Vite plugin that re-runs the generator on `buildStart` and watches for config changes in dev.

## Business Value
Without the generator, there's no mechanism to actually discover and wire up modules. Every new module would require manual registration — defeating the goal of a plug-in architecture. The Vite plugin integration means modules added during development are reflected without restarting the dev server.

## Acceptance Criteria
- [ ] `pnpm generate-modules` runs successfully and emits `src/core/module-registry.gen.ts`
- [ ] The generated file imports each discovered module's `nivora.config.ts` and exports `moduleRegistry: ModuleConfig[]`
- [ ] When a local module's `name` or `extends` matches an npm package, the generated file calls `deepMergeModuleConfig(baseConfig, localConfig)` and uses the merged result
- [ ] `pnpm dev` auto-regenerates the registry when any `src/modules/*/nivora.config.ts` changes (full HMR reload triggered)
- [ ] `pnpm build` triggers the generator before the Vite bundle step
- [ ] Generated file has a `THIS FILE IS AUTO-GENERATED` header and is listed in `.gitignore` alongside `routeTree.gen.ts`
- [ ] `pnpm lint` and `pnpm build` pass with zero errors

## Technical Design

### New npm scripts (package.json)
```json
"generate-modules": "tsx scripts/generate-modules.ts",
"collect-i18n": "tsx scripts/collect-i18n.ts"
```

### `scripts/generate-modules.ts` algorithm
```
1. Use fast-glob to find npm package configs:
   node_modules/@nivora-cms/*/nivora.config.{ts,js,mjs}

2. Use fast-glob to find local module configs:
   src/modules/*/nivora.config.ts

3. For each found config path, dynamically import() it (tsx handles TS):
   const mod = await import(resolvedPath)
   const config: ModuleConfig = mod.default

4. Build two maps:
   - npmModules: Map<name, { config, importPath }>
   - localModules: Map<name, { config, importPath, extendsName? }>

5. For each local module:
   - If config.name is in npmModules OR config.extends is in npmModules:
     → mark it as an override; the npm module is REMOVED from the registry
     → generate a merged import block in the output file
   - Otherwise: add as a new module

6. Write src/core/module-registry.gen.ts (see generated file shape below)
```

### Generated file shape (`src/core/module-registry.gen.ts`)
```ts
// THIS FILE IS AUTO-GENERATED BY scripts/generate-modules.ts
// DO NOT EDIT MANUALLY — run `pnpm generate-modules` to update

import type { ModuleConfig } from './module/index.ts'
import { deepMergeModuleConfig } from './module/merge.ts'

// ─── Module config imports ───────────────────────────────────────────────────
import _coreConfig from '../modules/core/nivora.config.ts'

// ─── Admin route factory imports ─────────────────────────────────────────────
import { adminRouteFactory as _coreAdminRouteFactory } from '../modules/core/routes/admin/index.ts'
import type { AnyRoute } from '@tanstack/react-router'

// ─── Registry ────────────────────────────────────────────────────────────────
export const moduleRegistry: ModuleConfig[] = [
  _coreConfig,
]

// ─── Admin routes ────────────────────────────────────────────────────────────
export const adminRouteFactories: Array<(parent: AnyRoute) => unknown> = [
  _coreAdminRouteFactory,
]

// ─── Helpers ─────────────────────────────────────────────────────────────────
export function getAllSidebarGroups() {
  return moduleRegistry
    .flatMap((m) => m.sidebar ?? [])
    .sort((a, b) => (a.order ?? 99) - (b.order ?? 99))
}

export function getAllSettingsFields() {
  return moduleRegistry.flatMap((m) => m.settings?.fields ?? [])
}

export function getModuleByName(name: string): ModuleConfig | undefined {
  return moduleRegistry.find((m) => m.name === name)
}
```

For a module that extends an npm package, the import block looks like:
```ts
import _blogBaseConfig from '@nivora-cms/blog/nivora.config.js'
import _blogLocalConfig from '../modules/blog/nivora.config.ts'
const _blogConfig = deepMergeModuleConfig(_blogBaseConfig, _blogLocalConfig)
// then _blogConfig is used in moduleRegistry instead of either raw config
```

### `vite/plugins/modules.ts`
```ts
import { execSync } from 'node:child_process'
import type { Plugin } from 'vite'

export function nivoraModulesPlugin(): Plugin {
  const run = () => execSync('pnpm generate-modules', { stdio: 'inherit' })
  return {
    name: 'nivora-modules',
    buildStart() { run() },
    configureServer(server) {
      server.watcher.add('src/modules/*/nivora.config.ts')
      server.watcher.on('change', (file) => {
        if (file.includes('nivora.config')) {
          run()
          server.hot.send({ type: 'full-reload' })
        }
      })
    },
  }
}
```

## Instructions for Developer
- `fast-glob` does not scan `node_modules` by default. Use `fg.glob('node_modules/@nivora-cms/*/nivora.config.{ts,js,mjs}', { dot: true, onlyFiles: true })` — the `dot: true` is needed for packages with dot-prefixed paths, and you must opt into `node_modules` scanning explicitly.
- Dynamic `import()` in the generator runs at Node.js level via `tsx`. Paths must be absolute: use `path.resolve(process.cwd(), relativePath)` before passing to `import()`.
- npm packages publish compiled JS (`.js` or `.mjs`), not `.ts`. The generator should try paths in this order: `.ts` (local), `.js`, `.mjs` (npm).
- The generated file's import paths must be relative to `src/core/`, not to `scripts/`. Use `path.relative` to compute correct relative paths from the output file's location.
- The generator should be idempotent — running it twice produces the same file. This is important for CI.
- Add `// THIS FILE IS AUTO-GENERATED` at the top AND a comment on the closing brace of each array so it's clear which block is which.
